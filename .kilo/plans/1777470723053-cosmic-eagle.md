# Integration Plan: Merging DM-Align Project with Unsloth Studio

> **Date**: 2026-04-29 | **Status**: Planning | **Author**: Kilo

---

## 1. Executive Summary

This plan maps the current `ml-lora-training` project onto **Unsloth Studio**, a local browser-based GUI for fine-tuning LLMs. The goal is to transition from a custom Docker-based CLI pipeline to leveraging Unsloth Studio's native UI for model training, data management, export, and chat evaluation — while preserving the project's core workflow: **synthetic data generation -> SFT training -> DPO training -> export -> evaluation**.

### Key Finding: Preference Optimization Gap (DPO, ORPO, GRPO)

**Unsloth Studio currently only supports SFT (QLoRA/LoRA/Full FT).** None of the preference optimization methods — DPO, ORPO, or GRPO — are exposed in the Studio UI. All three are supported by the Unsloth library itself via code/notebooks but require custom scripts. This means **any** alignment phase must remain as a custom workflow, even after Studio integration.

See Section 5 (DPO vs GRPO vs ORPO Analysis) for a detailed comparison of which method best fits this project.

## 3. What Is LoRA? (For First-Time Training Context)

Since you have not yet trained a LoRA, here is the essential conceptual framework:

### 3.1 The Problem LoRA Solves

A 27B-parameter model has ~27 billion weights. Full fine-tuning means updating all 27B weights, which requires:
- Storing gradients for all 27B parameters (~108GB+ VRAM in FP32)
- Storing optimizer states (~216GB+ VRAM for AdamW)
- This is impossible on a 32GB GPU

### 3.2 The LoRA Solution

**LoRA (Low-Rank Adaptation)** decomposes weight updates into two small matrices:

```
Instead of:  W_new = W_original + Delta_W  (Delta_W is 27B params)
LoRA does:   W_new = W_original + A * B    (A is rxd, B is dxr, where r << d)
```

With `r=32` and a layer dimension of `d=5120`:
- Full update: 5120 x 5120 = 26M parameters per layer
- LoRA update: 32 x 5120 + 5120 x 32 = 328K parameters per layer (**79x smaller**)

**Key insight**: The base model weights (`W_original`) stay frozen. Only the small `A` and `B` matrices are trained. After training, you can merge `A*B` back into `W_original` for a single deployable model.

### 3.3 QLoRA Extension

**QLoRA** adds 4-bit quantization of the base model, reducing VRAM from ~54GB (full 16-bit) to ~14GB (4-bit NF4). Combined with LoRA adapters, total VRAM fits in ~20-25GB — well within your RTX 5090's 32GB.

### 3.4 Why Unsloth Makes This Easier

Unsloth provides:
- **2x faster training** via optimized CUDA kernels
- **70% less VRAM** via memory-efficient implementations
- **Auto-configuration** of LoRA target modules, gradient checkpointing, and mixed precision
- **Studio UI** that handles all the boilerplate code you currently write manually

### Phase D: Preference Optimization (Remains Custom) — DPO vs GRPO vs ORPO Analysis

**Goal**: Determine the best preference optimization method for DM alignment, then implement as a custom script (no Studio UI support for any preference method).

#### Current State:
- `src/student/train_dpo.py` — Uses TRL `DPOTrainer` with Unsloth `PatchDPOTrainer()`
- `src/student/dpo_config.py` — DPO hyperparameters
- `src/teacher/generate_dpo_pairs.py` — Generates preference pairs
- `configs/dpo_config.yaml` — DPO config

#### Critical Finding: Studio Supports NO Preference Optimization Methods

| Method | Studio UI Support | Unsloth Library Support | Custom Script Needed |
|---|---|---|---|
| **DPO** (Direct Preference Optimization) | ❌ No | ✅ Yes | Yes |
| **ORPO** (Odds Ratio Preference Optimization) | ❌ No | ✅ Yes | Yes |
| **GRPO** (Group Relative Policy Optimization) | ❌ No | ✅ Yes | Yes |

All three methods require custom Python scripts or notebooks. The choice affects data format, training complexity, and expected results.

#### Comparison: DPO vs ORPO vs GRPO for DM Alignment

| Factor | DPO (Current) | ORPO | GRPO |
|---|---|---|---|
| **Data Format** | (chosen, rejected) pairs per prompt | (chosen, rejected) pairs per prompt | Questions + reward functions (no pairs needed) |
| **Concept** | Implicit reward from preference ratios | Combines SFT + DPO into single objective | RL with verifiable rewards, samples multiple responses per prompt |
| **VRAM Required** | ~20-25GB (QLoRA, 27B model) | Similar to DPO | Higher — needs multiple generations per prompt in memory |
| **Training Complexity** | Moderate — well-documented, stable | Moderate — single-stage, may be simpler | High — requires designing reward functions, tuning reward weights |
| **Data Preparation** | Generate chosen/rejected pairs (already done) | Same as DPO | Design reward functions (e.g., +1 for DM keywords, -1 for missing) |
| **Best For** | Fine-tuning alignment on known preferences | Single-stage alignment (skips separate SFT step) | Teaching reasoning, emergent behaviors, complex tasks |
| **Fit for DM Project** | **Good** — pairs already generated, well-understood | **Good** — could simplify pipeline (SFT+DPO combined) | **Less ideal** — DM alignment is about preference, not reasoning emergence |
| **Existing Code** | ✅ Already implemented | ❌ Requires rewrite | ❌ Requires significant rewrite |

#### Recommendation: Stay with DPO (for now)

**DPO remains the best choice** for this project because:
1. **Data is already prepared** — `generate_dpo_pairs.py` produces the correct format
2. **Code is already written** — `train_dpo.py` is tested and working (18/18 tests passing)
3. **Conceptually matches the goal** — DM alignment is about preferring certain responses over others, which is exactly what DPO optimizes
4. **Lower VRAM than GRPO** — GRPO requires holding multiple generations in memory simultaneously, which is tight on 32GB for a 27B model
5. **Proven pipeline** — SFT then DPO is the standard RLHF approach with predictable results

**ORPO is worth evaluating later** if DPO results are unsatisfactory. ORPO combines SFT and DPO into a single training stage, which could simplify the pipeline. However, it would require rewriting `train_dpo.py` and the data pipeline.

**GRPO is not recommended** for this project. GRPO excels at teaching reasoning capabilities (math, code, logic puzzles) where rewards are verifiable. DM alignment is about stylistic and ideological preference, not verifiable correctness — making DPO a better fit.

#### What Stays:
- **All DPO code remains unchanged** — Studio has no preference optimization UI
- DPO pair generation script remains unchanged
- DPO config files remain unchanged

#### What Changes:
1. **Input**: Instead of loading the SFT adapter from `checkpoints/lora_adapters/sft_adapter/`, load the **Studio training run checkpoint** (Studio saves LoRA adapters to `~/.unsloth/studio/` by default)
2. **Integration**: After SFT completes in Studio, export the LoRA adapter, then run DPO script pointing to that adapter
3. **Workflow**: Studio SFT -> Export LoRA -> Custom DPO script -> Export final adapter -> Studio Chat for evaluation

## 10. Appendix: ML Concepts Reference

### 10.1 Fine-Tuning Pipeline Stages

```
Pre-trained Model (e.g., Qwen3.5-27B-Instruct)
        │
        ▼
  ┌─────────────┐
  │   SFT       │  ← Teach the model YOUR domain (Dialectical Materialism)
  │ (Supervised  │     Input: 1,500 (question, DM-answer) pairs
  │ Fine-Tuning) │     Output: Model that generates DM-aligned responses
  └─────────────┘
        │
        ▼
  ┌─────────────┐
  │    DPO      │  ← Align the model's PREFERENCES to YOUR values
  │ (Direct     │     Input: 1,500 (question, chosen-DM, rejected-generic) pairs
  │ Preference  │     Output: Model that PREFERENCES DM-aligned responses
  │ Optimization)│     over generic ones, even when not explicitly prompted
  └─────────────┘
        │
        ▼
  Quantized GGUF Model (deployable, ~6-8GB on disk)
```

### 10.2 Why SFT Before DPO?

- **SFT** teaches the model *what to say* (capability)
- **DPO** teaches the model *what to prefer* (alignment)
- DPO requires the model to already understand the domain — running DPO on a non-SFT'd model produces poor results
- This two-stage approach is the standard RLHF (Reinforcement Learning from Human Feedback) pipeline

### 10.3 LoRA Hyperparameters Explained

- **Rank (r)**: Dimensionality of the low-rank decomposition. Higher = more capacity to learn, more VRAM. `r=32` is a good balance for 27B models.
- **Alpha**: Scaling factor for the LoRA update. Typically `alpha = r` or `alpha = 2*r`. Controls how aggressively the adapter modifies base weights.
- **Dropout**: Regularization to prevent overfitting. `0.05` is conservative — good for small datasets like yours (1,500 samples).
- **Target Modules**: Which layers to apply LoRA to. All 7 modules (attention + MLP) gives full coverage.

### 10.4 DPO Beta Parameter

- **Beta** controls the strength of the preference signal
- `beta=0.1` (your setting) is standard — moderate alignment strength
- Higher beta = stronger preference enforcement but risk of reducing model capability
- Lower beta = gentler alignment, may require more training steps
