## 1. Five Selected Benchmarks

Chosen for broad coverage of capability domains and direct comparability with published Qwen3.5-9B scores. All five are supported natively by lm-evaluation-harness.

| # | Benchmark | Domain | Task Type | Qwen3.5-9B Baseline | lm-eval Task Name |
|---|-----------|--------|-----------|---------------------|-------------------|
| 1 | **MMLU-Pro** | General Knowledge & Reasoning | Multiple-choice (cot) | **82.5** | `mmlu_pro` |
| 2 | **GPQA Diamond** | Scientific STEM | Multiple-choice | **81.7** | `gpqa:diamond` |
| 3 | **IFEval** | Instruction Following | Generation + rule checking | **91.5** | `ifeval` |
| 4 | **LiveCodeBench v6** | Code Generation | Generation | **65.6** | `livecodebench` |
| 5 | **HMMT** | Math Reasoning | Generation | **83.2** (Feb 25) / **82.9** (Nov 25) | `hmmt` |

### Why These Five?

- **MMLU-Pro**: Successor to MMLU; harder adversarial distractors, 10k questions across 64 subjects. The de-facto standard knowledge benchmark for 7-10B models.
- **GPQA Diamond**: Expert-annotated graduate-level science questions. Best single benchmark for STEM depth.
- **IFEval**: Instruction-following accuracy. Critical regression test — fine-tuning on DM-aligned data should NOT degrade instruction compliance.
- **LiveCodeBench v6**: Time-locked code problems (prevents contamination). Tests whether LoRA adaptation broke code generation.
- **HMMT**: Math competition problems. Tests quantitative reasoning preservation. (GSM8K was considered but is not published for Qwen3.5-9B; HMMT is the published math benchmark and is harder/more discriminating.)

### Note on GSM8K

GSM8K was considered but excluded — not published for Qwen3.5-9B, and HMMT provides harder/more discriminating math coverage with a published baseline.

## 2. Evaluation Architecture

### 2.1 Environment: Local WSL2 (Not Docker)

All evaluation runs execute directly on WSL2 with GPU passthrough through Docker Desktop's NVIDIA runtime. The RTX 5090 (32GB) is accessible from WSL2 via the NVIDIA container toolkit bridge.

**Prerequisites**:
- Python 3.10+ virtual environment in WSL2 (current system Python is 3.9.12 — we'll create a venv)
- PyTorch with CUDA 12.6 support
- lm-evaluation-harness with `hf` backend
- CUDA-visible GPU confirmed via `nvidia-smi` from WSL2 (already working)

**Why local, not Docker**: Evaluation is inference-only (no training). Running locally avoids container overhead, simplifies path handling for GGUF files on the Windows filesystem, and gives direct access to the GPU without Docker GPU passthrough layers.

### 2.2 Framework: lm-evaluation-harness

EleutherAI's lm-evaluation-harness is the industry standard backend (used by HuggingFace Open LLM Leaderboard). It supports:
- GGUF models via the `hf` backend (with separate tokenizer to avoid hours-long tokenizer reconstruction)
- LoRA/PEFT adapters via `peft=` model arg
- All five selected benchmarks natively
- JSON result output for comparison

### 2.3 Model Loading Strategy

Three models available for evaluation:

| Model | Location | Format | Size |
|-------|----------|--------|------|
| **BF16 Baseline** | HF cache `models--Qwen--Qwen3.5-9B/snapshots/c202236235762e1c871ad0ccb60c8ee5ba337b9a/` | 4× safetensors shards | ~18 GB |
| **Q4_K_M Baseline** | HF cache `models--unsloth--Qwen3.5-9B-GGUF/snapshots/3885219b6810b007914f3a7950a8d1b469d598a5/Qwen3.5-9B-Q4_K_M.gguf` | GGUF | 5.3 GB |
| **Fine-tuned Q4_K_M** | Studio export `C:\Users\Guy\.unsloth\studio\exports\Qwen3.5-9B-gguf\` | GGUF | ~5.3 GB |

For GGUF models, the tokenizer must be passed separately to avoid hanging during tokenizer reconstruction:

```
lm_eval --model hf \
  --model_args pretrained=/path/to/model,gguf_file=filename.gguf,tokenizer=Qwen/Qwen3.5-9B \
  --tasks mmlu_pro,gpqa:diamond,ifeval,livecodebench,hmmt \
  --batch_size auto:4 \
  --output_path evals/results/runs/finetuned \
  --log_samples
```

### 2.4 Three-Way Comparison Strategy

The scientifically clean approach isolates three effects:

| Comparison | What It Measures |
|------------|-----------------|
| **BF16 baseline vs Published BF16 scores** | Validates evaluation framework correctness |
| **BF16 baseline vs Q4_K_M baseline** | Quantifies quantization penalty alone |
| **Q4_K_M baseline vs Q4_K_M fine-tuned** | Isolates fine-tuning effect (PRIMARY metric) |

**Recommended run order**:

| # | Run | Model Path | Purpose |
|---|-----|------------|---------|
| 1 | Fine-tuned Q4 | `/mnt/c/Users/Guy/.unsloth/studio/exports/Qwen3.5-9B-gguf/` | Primary target (~10-15 min, most headroom) |
| 2 | Baseline Q4 | HF cache `unsloth/Qwen3.5-9B-GGUF` snapshot | Quantized baseline (~10-15 min, no quantization needed) |
| 3 | Baseline BF16 | HF cache `Qwen/Qwen3.5-9B` snapshot | Framework validation (~20-35 min, tight VRAM) |

**Interpretation**:
- Run 3 vs Published: Should be within ±1 point. If not, eval harness differs from official setup.
- Run 3 vs Run 2: Shows quantization penalty. Expect 1-3 point drop per benchmark.
- Run 2 vs Run 1: **The number that matters.** Isolates fine-tuning effect without quantization noise.

**Key advantage**: The Q4_K_M baseline GGUF is already pre-quantized and cached locally — no `llama.cpp/quantize` step needed.

## 4. VS Code Workspace Integration for Unsloth Directory

### Problem

You want visibility into `C:\Users\Guy\.unsloth\` (Studio exports, llama.cpp configs, etc.) from VS Code without:
- Connecting to the Git repo inside `llama.cpp/` (which has its own `.git`)
- Polluting the project's source control view

### Solution: Multi-Root Workspace with Git Exclusion

Create a VS Code workspace file that adds the Unsloth directory as a **second root** with git ignored.

**File**: `.vscode/ml-lora-training.code-workspace`

```json
{
  "folders": [
    {
      "name": "ml-lora-training",
      "path": "."
    },
    {
      "name": "unsloth-external",
      "path": "/mnt/c/Users/Guy/.unsloth"
    }
  ],
  "settings": {
    "files.exclude": {
      "**/node_modules": true,
      "**/*.pyc": true
    },
    "search.exclude": {
      "/mnt/c/Users/Guy/.unsloth/studio/.venv_t5_530/**": true,
      "/mnt/c/Users/Guy/.unsloth/llama.cpp/build/**": true,
      "/mnt/c/Users/Guy/.unsloth/llama.cpp/__pycache__/**": true,
      "/mnt/c/Users/Guy/.unsloth/.cache/**": true
    }
  }
}
```

**Key settings explained**:

| Setting | Purpose |
|---------|---------|
| `folders` with two entries | Multi-root workspace: project + Unsloth |
| `"name": "unsloth-external"` | Labels the second root in Explorer sidebar |
| `search.exclude` | Excludes venv, build artifacts, cache from search/indexing |
| `files.exclude` | Hides large/irrelevant files from Explorer |

### Preventing Git Integration

VS Code's Source Control view auto-detects `.git` folders. To prevent the `llama.cpp/.git` from appearing:

**Add to workspace settings**:
```json
{
  "git.enabled": false,
  "git.ignoreLegacyWarning": true
}
```

Or more selectively, use the `git.ignoreRepositories` setting:
```json
{
  "git.ignoreRepositories": [
    "/mnt/c/Users/Guy/.unsloth/llama.cpp"
  ]
}
```

This keeps git enabled for the main project but ignores the llama.cpp repo.

### Also Create: Symbolic Links for Quick Access

In addition to the workspace file, create symlinks for lightweight access to frequently-needed directories:

```bash
mkdir -p references
ln -s /mnt/c/Users/Guy/.unsloth/studio/exports references/unsloth-exports
ln -s /mnt/c/Users/Guy/.unsloth/studio/models references/unsloth-models
```

Add to `.gitignore`:

This gives you quick access to exports and models directly within the project tree without the full Unsloth directory structure.

## 5. VRAM Allocation Analysis

### 5.1 GPU Resource Budget

| Resource | Value |
|----------|-------|
| **GPU** | NVIDIA RTX 5090 |
| **Total VRAM** | 32,607 MB (~31.8 GB) |
| **Reserved (driver/desktop)** | ~1-2 GB |

### 5.2 Fine-Tuned GGUF Model (Q4_K_M) — Actual Run

The fine-tuned model is the Q4_K_M GGUF (~5.3 GB). VRAM breakdown:

| Component | Calculation | VRAM |
|-----------|-------------|------|
| **Model weights** | Q4_K_M GGUF | **~5.3 GB** |
| **KV cache** | 32 layers × 4 KV heads × 256 dim × 4-bit × 16K tokens | **~128 MB** |
| **Framework overhead** | PyTorch activations, temp buffers, lm-eval state | **~2-3 GB** |
| **Batch buffers** | Depends on batch_size (auto:4 will find optimal) | **~2-4 GB** |
| **Total estimated** | | **~9.5-11.5 GB** |
| **Headroom remaining** | 30 GB - 11.5 GB | **~18.5 GB** |

**Context window per benchmark**:

| Benchmark | Max Input Length | Max Output Length | Total Context | KV Cache |
|-----------|-----------------|-------------------|---------------|----------|
| MMLU-Pro | ~1K (MC question) | ~2K (CoT) | ~3K | ~24 MB |
| GPQA Diamond | ~1K | ~2K | ~3K | ~24 MB |
| IFEval | ~2K | ~1K | ~3K | ~24 MB |
| LiveCodeBench | ~3K | ~4K | ~7K | ~56 MB |
| HMMT | ~1K | ~8K (reasoning) | ~9K | ~72 MB |

The fine-tuned GGUF model has massive headroom. Batch size will be the limiting factor, not VRAM.

### 5.3 Baseline BF16 Model — Full Precision Run

The baseline `Qwen/Qwen3.5-9B` loads in BF16 (~18 GB). VRAM breakdown:

| Component | Calculation | VRAM |
|-----------|-------------|------|
| **Model weights** | 9B params × 2 bytes (BF16) | **~18.0 GB** |
| **KV cache** | 32 layers × 4 KV heads × 256 dim × 16-bit × 16K tokens | **~256 MB** |
| **Framework overhead** | PyTorch activations, temp buffers | **~3-4 GB** |
| **Batch buffers** | Depends on batch_size | **~2-4 GB** |
| **Total estimated** | | **~23.5-26.5 GB** |
| **Headroom remaining** | 30 GB - 26.5 GB | **~3.5-6.5 GB** |

The BF16 baseline fits but with tight headroom. Batch size will be constrained (~4-8).

### 5.4 Ideal Model + Context from Remaining VRAM

**With GGUF fine-tuned model (~11.5 GB used, ~18.5 GB free):**
- Can comfortably run at batch_size 32+ with 16K context
- Could simultaneously hold a second model in memory for comparison
- Could upgrade to Qwen3.5-14B Q4_K_M (~9 GB) and still have ~12 GB headroom

**With BF16 baseline (~26.5 GB used, ~3.5-6.5 GB free):**
- Batch size limited to ~4-8
- Context window limited to ~8-16K before OOM
- Cannot hold two models simultaneously

**Recommendation**: Run the fine-tuned GGUF evaluation first (more headroom, faster), then the BF16 baseline (tighter but still fits).

## 6. Expected Outcomes

### Quantization Penalty (BF16 → Q4_K_M baseline)

Expect 1-3 point drop per benchmark from quantization alone:

| Benchmark | BF16 Baseline | Q4 Baseline (Expected) | Quantization Penalty |
|-----------|--------------|------------------------|---------------------|
| MMLU-Pro | ~82.5 | ~80-82 | ~1-2 pts |
| GPQA Diamond | ~81.7 | ~79-81 | ~1-2 pts |
| IFEval | ~91.5 | ~89-91 | ~1-2 pts |
| LiveCodeBench | ~65.6 | ~63-65 | ~1-2 pts |
| HMMT | ~83.2 | ~81-83 | ~1-2 pts |

### Fine-Tuning Effect (Q4 baseline → Q4 fine-tuned)

**Best Case (No Regression)**:
Fine-tuned model scores within ±1 point of Q4 baseline on all five benchmarks, confirming the DM alignment training did not degrade general capabilities.

**Acceptable (Minor Regression)**:
- IFEval drops ≤ 3 points (instruction following slightly weakened by domain shift)
- MMLU-Pro drops ≤ 2 points (knowledge intact)
- Code/Math benchmarks stable (domain-specific reasoning preserved)

**Concerning (Significant Regression)**:
- Any benchmark drops > 5 points from Q4 baseline
- IFEval drops > 5 points (model no longer follows formatting instructions)
- LiveCodeBench drops > 5 points (code generation capability degraded)

If significant regression detected, the DPO training may need:
- More negative data pairs (neutral-domain reinforcement)
- Lower learning rate
- Earlier stopping based on eval checkpoint
