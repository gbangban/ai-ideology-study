# Plan: Run GGUF Models in lm-eval-harness

## Context

- **GGUF file**: `checkpoints/base_model/Qwen3.5-27B-Instruct-Q4_K_M.gguf` (~16 GB, qwen35 architecture)
- **GPU**: RTX 5090 (32 GB VRAM), CUDA 13.0/13.2
- **llama.cpp server**: Available at `C:\llamacpp\llama-server.exe` (CUDA build, accessible via WSL2 at `/mnt/c/llamacpp/`)
- **Current eval scripts**: All use `--model hf` with safetensors. GGUF not evaluated yet.
- **lm-eval version**: 0.4.12 (has `gguf` backend registered as `GGUFLM`)

## Problem

The current eval scripts use `--model hf` which loads safetensors via transformers. The GGUF model cannot be loaded this way. We need to evaluate the Q4_K_M GGUF baseline to measure quantization penalty.

## Approach: llama.cpp server + lm_eval `gguf` backend

The `gguf` backend (`GGUFLM`) in lm-eval is an HTTP client that connects to a running OpenAI-compatible API server at `{base_url}/v1/completions`. It does NOT load GGUF files directly.

We'll use the existing `llama-server.exe` from `C:\llamacpp` as the backend server, managed by a new eval script.

## Implementation

### 1. Create `evals/scripts/run_baseline_gguf.sh`

New script based on the existing `run_baseline_q4.sh` template. Key differences:

| Aspect | Current Scripts | New GGUF Script |
|--------|----------------|-----------------|
| Model backend | `--model hf` | `--model gguf` |
| Model args | `pretrained=<dir>,dtype=bfloat16` | `base_url=http://127.0.0.1:8080,max_length=4096` |
| Model source | HF safetensors cache | GGUF file path |
| Server | N/A | llama-server.exe (auto-managed) |
| Results dir | `results/baseline/q4/` or `results/baseline/bf16/` | `results/baseline/gguf/` |

### 2. Server management

The script manages the llama-server lifecycle:

```bash
# Configuration
LLAMA_SERVER="/mnt/c/llamacpp/llama-server.exe"
GGUF_PATH="${GGUF_MODEL_PATH:-checkpoints/base_model/Qwen3.5-27B-Instruct-Q4_K_M.gguf}"
SERVER_PORT="${GGUF_SERVER_PORT:-8080}"
SERVER_CTX="${GGUF_CTX_SIZE:-4096}"

# Start server (background)
$LLAMA_SERVER \
  --model "$GGUF_PATH" \
  --host 127.0.0.1 \
  --port "$SERVER_PORT" \
  --n-gpu-layers all \
  --ctx-size "$SERVER_CTX" \
  --batch-size 2048 \
  --flash-attn \
  --logprobs 10 \
  > "$RESULTS_DIR/server.log" 2>&1 &
SERVER_PID=$!

# Poll server until ready
for i in $(seq 1 30); do
  curl -s "http://127.0.0.1:$SERVER_PORT/health" >/dev/null 2>&1 && break
  sleep 1
done

# Cleanup trap
trap 'kill $SERVER_PID 2>/dev/null; wait $SERVER_PID 2>/dev/null' EXIT INT TERM
```

### 3. Eval command per task

```bash
lm_eval --model gguf \
  --model_args base_url=http://127.0.0.1:$SERVER_PORT,max_length=$SERVER_CTX \
  --tasks "$TASK" \
  --batch_size 4 \
  --output_path "$RESULTS_DIR" \
  --log_samples \
  2>&1 | tee -a "$EVAL_LOG"
```

### 4. Server configuration rationale

| Parameter | Value | Reason |
|-----------|-------|--------|
| `--n-gpu-layers all` | All layers on GPU | 27B Q4 (~16GB) fits in 32GB VRAM |
| `--ctx-size 4096` | 4096 | Sufficient for all eval tasks |
| `--batch-size 2048` | 2048 | Large logical batch for prompt processing |
| `--flash-attn` | enabled | Faster attention on RTX 5090 |
| `--logprobs 10` | 10 | Matches GGUFLM's hardcoded `self.logprobs = 10` |

### 6. Known limitation: `loglikelihood_rolling`

The GGUF backend raises `NotImplementedError` for `loglikelihood_rolling`. If any task requires it, the script will:
- Catch the failure gracefully
- Log a warning
- Continue with remaining tasks
- Report which tasks were skipped due to this limitation

### 7. Script features (consistent with existing scripts)

- `--tasks TASK1,TASK2` — run specific tasks
- `--dry-run` — show what would run
- `--help` — show usage
- `FORCE_RERUN=true` — skip existing results check
- `AUTO_CONTINUE=true` — skip VRAM threshold prompt
- Results cached per task (`<task>.json`) to support resume
- Server log saved alongside results
