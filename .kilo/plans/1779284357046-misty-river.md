# Plan: Run GGUF Models in lm-eval-harness

## Context

- **GGUF file**: `checkpoints/base_model/Qwen3.5-27B-Instruct-Q4_K_M.gguf` (~16 GB)
- **GPU**: RTX 5090 (32 GB VRAM), CUDA 13.0/13.2
- **Current eval scripts**: All use `--model hf` with safetensors (bfloat16). GGUF is not used yet.
- **lm-eval version**: 0.4.12

## Problem

The current eval scripts use `--model hf` which loads safetensors via transformers. The GGUF model cannot be loaded this way. We need to evaluate the Q4_K_M GGUF baseline.

## Available Approaches

### Option A: llama-cpp-python server + lm_eval `gguf` backend (Recommended)

The `gguf` model backend in lm-eval (`GGUFLM`) is an HTTP client that connects to a running **OpenAI-compatible API server**. It does NOT load GGUF files directly.

**Setup:**
1. Install `llama-cpp-python` with CUDA support in the evals venv
2. Start `llama-cpp-python` as a server serving the GGUF file on localhost
3. Point lm_eval `--model gguf` at the server's `base_url`

**Command flow:**
```bash
# Start server (background)
python3 -m llama_cpp.server --model path/to/model.gguf --host 0.0.0.0 --port 8080 --n-gpu-layers -1

# Run eval
lm_eval --model gguf --model_args base_url=http://localhost:8080 --tasks humaneval ...
```

**Pros:**
- Uses the GGUF file directly (no conversion needed)
- Evaluates the actual quantized model (measures quantization penalty)
- Low overhead — llama-cpp-python is lightweight
- GPU inference via CUDA backend

**Cons:**
- Requires managing a background server process
- Slower batch inference than vllm (no PagedAttention)
- `loglikelihood_rolling` not supported by the gguf backend

### Option B: vllm with GGUF + lm_eval `vllm` backend

vllm natively supports GGUF files and has a built-in lm_eval backend (`--model vllm`).

**Setup:**
1. Install `vllm` in the evals venv (`pip install vllm`)
2. Run `lm_eval --model vllm --model_args pretrained=path/to/model.gguf ...`

**Pros:**
- Fastest inference (PagedAttention, continuous batching)
- Single command, no server management
- Full lm_eval feature support

**Cons:**
- vllm is a large install (~2-3 GB additional dependencies)
- May have compatibility issues with `qwen35` architecture (newer than most vllm releases)
- vllm's GGUF support may not fully support all Q4_K_M tensor types for qwen35

### Option C: llama.cpp binary server (external)

Build/install llama.cpp from source, run `llama-server`, connect lm_eval to it.

**Cons:** More complex build, essentially same as Option A but with more setup steps.

## Recommendation: Option A (llama-cpp-python server)

**Rationale:**
- Lowest risk — llama-cpp-python handles Q4_K_M quantization natively and is the reference implementation for GGUF
- Smaller install footprint vs vllm
- The `qwen35` architecture is newer; llama-cpp-python gets architecture updates faster than vllm's GGUF loader
- We already have the `gguf` backend registered in lm_eval (just needs the server)
- For a 27B Q4 model on 32 GB VRAM, llama-cpp-python with CUDA offloads all layers to GPU

## Implementation Plan

### 1. Install llama-cpp-python with CUDA

```bash
cd evals
source .venv/bin/activate
CMAKE_ARGS="-DGGML_CUDA=on" pip install llama-cpp-python --force-reinstall --no-cache-dir
```

### 2. Create new eval script: `evals/scripts/run_baseline_gguf.sh`

Based on the existing `run_baseline_q4.sh` template, but:
- Uses `--model gguf` with `base_url` pointing to llama-cpp-python server
- Manages the server lifecycle (start before evals, kill after)
- Points to the GGUF file at `checkpoints/base_model/Qwen3.5-27B-Instruct-Q4_K_M.gguf`
- Results go to `evals/results/baseline/gguf/`
- Same task suite: `mmlu_pro,gpqa_diamond_zeroshot,ifeval,humaneval,leaderboard_math_hard`

### 3. Server management in the script

```bash
# Start server
python3 -m llama_cpp.server \
  --model "$GGUF_PATH" \
  --host 127.0.0.1 \
  --port 8080 \
  --n-gpu-layers -1 \
  --ctx-size 4096 \
  --batch-size 2048 \
  --logprobs 10 \
  --embedding-batch-size 2048 \
  > "$RESULTS_DIR/server.log" 2>&1 &
SERVER_PID=$!

# Wait for server ready
sleep 5  # or poll the /v1/models endpoint

# Run evals
lm_eval --model gguf \
  --model_args base_url=http://127.0.0.1:8080,max_length=4096 \
  --tasks "$TASK" \
  --batch_size 4 \
  ...

# Cleanup
kill $SERVER_PID 2>/dev/null
```

### 4. Key configuration

| Parameter | Value | Reason |
|-----------|-------|--------|
| `--n-gpu-layers -1` | All layers on GPU | 27B Q4 fits in 32 GB |
| `--ctx-size 4096` | 4096 | Sufficient for eval tasks |
| `--logprobs 10` | 10 | Matches GGUFLM's expected logprobs |
| `max_length=4096` | 4096 | Matches ctx-size |

### 5. Note on `loglikelihood_rolling`

The GGUF backend raises `NotImplementedError` for `loglikelihood_rolling`. Tasks that require it (e.g., `leaderboard_math_hard` may use it) will fail. We'll need to:
- Test each task and handle failures gracefully
- For tasks requiring `loglikelihood_rolling`, either skip them or fall back to the `hf` backend

### 6. Script structure (mirrors existing scripts)

```
evals/scripts/run_baseline_gguf.sh
  - Sources eval_logging.sh
  - Same --tasks, --dry-run, --help flags
  - GGUF_MODEL_PATH env var (defaults to checkpoints/base_model/)
  - Server auto-start/stop
  - Results to evals/results/baseline/gguf/
  - Skips tasks that require loglikelihood_rolling
```
