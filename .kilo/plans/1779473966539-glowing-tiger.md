# Plan: Integrate EconCausal and Corr2Cause Evaluations End-to-End

## Overview

Add two custom causal reasoning benchmarks to the existing BF16 eval pipeline:
- **EconCausal**: 10,490 context-annotated causal triplets from economics/finance papers. 4-class classification (`+`, `-`, `None`, `mixed`). Three subtasks with increasing difficulty.
- **Corr2Cause**: ~1.2K test samples of pure causal inference from statistical correlations. Binary classification (`True`/`False`).

Both will use `lm-evaluation-harness` custom YAML task definitions loaded via `--include_path`, running in BF16 against both baseline and finetuned models.

## Dataset Format Summary

### EconCausal
- Source: `https://github.com/econaikaist/econcausal-benchmark` (CC BY-NC 4.0)
- Format: JSONL with `question` (full prompt text) and `answer` (one of `+`, `-`, `None`, `mixed`)
- Data files: `data/tasks/task1_econ.jsonl`, `task1_finance.jsonl`, `task2.jsonl`, `task3.jsonl`
- Task 1: Causal sign prediction with context (947 econ + 860 finance)
- Task 2: Context-dependent sign prediction (284 instances)
- Task 3: Misinformation-robust sign prediction (852 instances)

### Corr2Cause
- Source: `https://huggingface.co/datasets/causal-nlp/corr2cause`
- Format: CSV with `input` (premise + hypothesis text), `label` (0/1), `num_variables`, `template`
- Splits: train (206K), validation (1.08K), test (1.16K)
- We evaluate on the `test` split only

## Implementation Steps

### Step 1: Create custom task Python modules

**File: `evals/src/econcausal.py`**
- Custom lm-eval task class for EconCausal using `Task` base class
- Loads JSONL data from GitHub raw URLs (or downloads to `evals/data/` on first run)
- Generates prompts from the `question` field, extracts answer from `answer` field
- Uses `multiple_choice` generation mode with 4 choices: `+`, `-`, `None`, `mixed`
- Metric: `acc` (accuracy)
- Subtasks: `econcausal_task1_econ`, `econcausal_task1_finance`, `econcausal_task2`, `econcausal_task3`

**File: `evals/src/corr2cause.py`**
- Custom lm-eval task class for Corr2Cause
- Loads from HF dataset `causal-nlp/corr2cause` split=`test` (already available via `datasets` lib)
- Formats as a binary choice: the model gets the `input` field and must choose between `True` and `False`
- Metric: `acc` (accuracy)

**File: `evals/src/__init__.py`** (empty, makes `src` a package)

## Technical Details

### EconCausal approach
The JSONL already contains full prompt text in `question` and the answer in `answer`. The cleanest approach is a **Python task class** (`evalsrc/econcausal.py`) that:
- Downloads the JSONL from GitHub raw on first load (cached in `evals/data/econcausal/`)
- Uses `output_type: generate_until` with a simple regex to extract `predicted_sign` from the JSON output, OR
- Uses `output_type: multiple_choice` with the 4 sign options

Given the prompts already include the instruction to output JSON with `predicted_sign`, the multiple_choice approach won't work directly since the model generates free text. **Better approach**: use `output_type: generate_until` with a custom `process_results` method that extracts the sign from the model's JSON output using regex.

### Corr2Cause approach
Similarly, the model needs to reason about the premise then answer True/False. Use `output_type: generate_until` with a prompt template that appends a classification instruction, then parse with regex.

## Key Design Decisions

1. **Python task classes over pure YAML** — Both datasets require custom prompt formatting and answer parsing (JSON extraction for EconCausal, True/False regex for Corr2Cause). Python `Task` subclasses with `process_results()` give full control.

2. **Data download on first run** — EconCausal data is downloaded from GitHub raw URLs on first evaluation run, cached in `evals/data/econcausal/`. Corr2Cause loads via `datasets` library from HF.

3. **`--include_path` for both YAML and Python** — The scripts will pass `--include_path evals/configs/task_configs:evals/src` so lm-eval discovers both the YAML configs and Python task modules.

4. **BF16 only** — Consistent with project constraints. No GGUF evals for these tasks.

5. **Results directory structure** — Results saved alongside existing results: `evals/results/baseline/bf16/` and `evals/results/finetuned/bf16/`.

6. **EconCausal data** — Download from GitHub raw URLs on first run, cached in `evals/data/econcausal/`. All 4 subtasks included (Task1 Econ, Task1 Finance, Task2, Task3).
