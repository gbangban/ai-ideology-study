# Fix Broken Evals: Disable Thinking Mode for Qwen3.5-9B Instruct

## Problem

All custom eval tasks (corr2cause, econcausal_*) return **0% accuracy** because Qwen3.5-9B Instruct has a native thinking mode that's always enabled in the chat template. When `apply_chat_template(add_generation_prompt=True)` is called, it appends `\n
</think>

\n\n` (the thinking opening tag), causing the model to always start generating with `Thinking Process:`. The `until` stop sequence includes `\n`, so generation stops immediately after `Thinking Process:` — the model never reaches the actual answer.

**Evidence from latest runs (2026-05-22):**
- `corr2cause` baseline: `acc = 0` (all responses: `Thinking Process:`)
- `corr2cause` finetuned: `acc = 0` (all responses: `Thinking Process:`)
- Both models produce identical broken output

**Root cause:** The `--apply_chat_template` flag in the eval scripts triggers the thinking mode in the chat template. lm_eval 0.4.12 supports `enable_thinking=False` via model args, which strips the thinking block.

## Solution

Add `enable_thinking=False` to the model args in both eval scripts. This disables the thinking/reasoning mode in the chat template, so the model generates direct answers instead of `Thinking Process:` traces.

### Changes

#### 1. `evals/scripts/run_baseline_bf16.sh` — Add `enable_thinking=False` to model args

Change the `lm_eval` command (line ~183):
```bash
# Before:
lm_eval --model hf \
  --model_args pretrained=$MODEL_DIR,dtype=bfloat16 \

# After:
lm_eval --model hf \
  --model_args pretrained=$MODEL_DIR,dtype=bfloat16,enable_thinking=False \
```

## Why This Works

lm_eval 0.4.12's `HFLM` class accepts `enable_thinking` as a model arg. When set to `False`, it passes `enable_thinking=False` to `tokenizer.apply_chat_template()`, which tells the Qwen3.5 chat template to skip the `\n
</think>

\n\n` thinking block. The model then generates direct answers without the thinking preamble.

This affects:
- **corr2cause** — will generate `True` or `False` directly instead of `Thinking Process:`
- **econcausal_*** — will generate JSON directly instead of `Thinking Process:`
- **Standard tasks** (humaneval, ifeval, mmlu, etc.) — these already work because their stop sequences and generation limits let the model complete past the thinking block, but disabling thinking will make them faster and more reliable

## Risk Assessment

**Low risk.** This is a single parameter change that affects only the chat template behavior. The model weights, tokenizer, and task configs remain unchanged. The `enable_thinking=False` flag is specifically designed by lm_eval for this use case with reasoning models like Qwen3.5.
