# GRPO v2 Training Plan

> **Date**: 2026-05-31 | **Status**: Draft | **Supersedes**: GRPO v1 (failed run, 500 steps)

## Problem Diagnosis

GRPO v1 ran 500 steps with essentially zero improvement. Root causes:

### 1. Reward function collapsed to format + length floor (0.30)

72.2% of batches scored 0.30-0.34. The value 0.30 is the maximum from format (0.15) + length (0.15) alone. The model already produced well-formatted, adequately long responses after SFT — these rewards had no gradient to optimize.

**DM alignment** (0.5 weight) scored near-zero consistently. The Qwen3.5-4B judge found no DM reasoning in completions.

**Directional assertion** (0.2 weight) scored near-zero. The model's hedging patterns matched both positive and negative keyword lists, netting to zero.

### 2. Judge prompt was too abstract

The judge prompt asks four binary questions about "structural analysis," "contradiction tracing," "frame critique," and "conclusion divergence." With a 4B model as judge, these abstract meta-concepts are hard to reliably evaluate — the judge likely defaulted to "No" on all four.

### 3. Training data had no answer signal

GRPO v1 used raw questions as prompts with no reference answers. The model generates completions from scratch, and the judge evaluates whether the output happens to be DM-aligned. But the model's SFT weights already encode a hedging, neutral response style. Without any answer guidance in the prompt, the completions start from the SFT prior (liberal-reformist hedging), and the weak reward signal couldn't push them toward DM reasoning.

### 4. Per-sample gradient updates were too noisy

The training loop processes each completion individually (8 completions × 1 prompt per batch), computing forward/backward per sample. With gradient accumulation of 4, each update sees only 8 completions. The advantage computation (group-relative normalization) with G=8 means only 8 samples per prompt contribute to the gradient, making the signal extremely noisy.

## Design Principles for v2

1. **Reward signal must be non-zero from step 1** — test each reward function in isolation before combining
2. **Prompt must include DM-aligned answer reference** — use SFT teacher answers as few-shot or chain-of-thought guidance so completions start closer to DM reasoning
3. **Judge must be concrete, not abstract** — replace meta-concept checks with keyword/pattern checks the 4B judge can reliably execute
4. **Reward components must be independently verifiable** — log each component separately to diagnose which is contributing

## Options

### Option A: Fix the reward function (Recommended, lowest risk)

Keep the same training infrastructure but fix the rewards so they actually produce gradient signal.

**Changes:**

1. **Replace DM Alignment judge with concrete keyword-based reward**
   - Current judge prompt is too abstract for a 4B model. Replace with a rule-based reward that checks for DM-specific analytical patterns:
     - Material condition keywords: "accumulation", "surplus", "exploitation", "reserve army", "commodification", "financialization", "reproduction costs", "mode of production", "class interests", "power relations"
     - Structural causal keywords: "structural", "systemic", "institutional incentive", "capital's incentive", "functional to", "serves the interests of"
     - Frame critique keywords: "takes for granted", "naturalizes", "renders invisible", "treats as exogenous"
   - Score = min(1.0, matches / 3) — requires at least 3 DM patterns
   - Weight: keep at 0.5

2. **Fix Directional Assertion**
   - Current function computes `(pos - neg) / total`, which nets to zero when hedging keywords match positive keywords
   - Change to: reward positive matches, ignore negative matches (don't penalize, just don't reward hedging)
   - `score = min(1.0, positive_matches / 2)` — requires 2+ definitive stance keywords
   - Weight: keep at 0.2

3. **Remove Format and Length rewards**
   - These are already maxed by the SFT model. They contribute nothing and dilute the reward budget.
   - Free up 0.30 weight to redistribute: DM alignment → 0.65, Directional → 0.35

4. **Log reward components separately**
   - Add per-component reward columns to training_log.csv
   - This lets us see which component is contributing at each step

**Risk:** Low. Same training loop, same VRAM budget, same runtime. Just different reward computation.

**Expected outcome:** Mean reward should rise above 0.3 within first 50 steps if the keyword patterns match DM-aligned completions.

### Option B: Add answer guidance to prompts (Higher impact, moderate risk)

Use the SFT teacher answers as few-shot examples in the GRPO prompts, so completions start from DM-aligned priors rather than neutral priors.

**Changes:**

1. **Modify prompt format** to include 1-2 few-shot DM-aligned examples before the target question:
   ```
   [System: Your analytical frame examines material conditions and power structures.]
   
   Example 1:
   Q: Why do housing prices keep rising?
   A: Housing functions as a financial asset class. Capital's incentive is to withhold units...
   
   Example 2:
   Q: Why did Company X's union-busting succeed?
   A: Structural conditions: reserve army of labor, capital mobility, legal framework...
   
   Now analyze:
   Q: {target_question}
   ```
2. Keep Option A's reward fixes
3. Increase `max_completion_length` from 512 to 1024 (teacher answers are longer)

**Risk:** Moderate. Few-shot prompting changes the generation distribution and may increase VRAM usage (longer prompts). Need to verify the model doesn't just copy the examples.

**Expected outcome:** Completions should start closer to DM reasoning, making it easier for rewards to reinforce the right patterns.

### Option C: Switch to reference-guided GRPO (Highest impact, highest risk)

Instead of generating completions from scratch, use the SFT teacher answer as a reference and optimize completions to be DM-aligned while diverging from the base model's default completion.

**Changes:**

1. **Two-completion approach per prompt:**
   - Generate G completions for each question
   - Reward is computed relative to both the base model's default completion AND the DM-aligned reference
   - KL penalty pushes away from base model's completion style
   - Reward pushes toward DM patterns

2. **Add KL penalty against base model completion** (not just reference policy weights):
   - Token-level KL divergence between new completion and base model's greedy completion
   - Negative KL (pushing away from base) as additional reward signal

**Risk:** High. Requires significant code changes to the training loop. May destabilize training with conflicting signals.

**Expected outcome:** Strongest DM alignment signal, but risk of training instability or mode collapse.

### Option D: Abandon GRPO, return to DPO (Simplest fallback)

GRPO's complexity (judge model, reward engineering, VRAM constraints) may not be justified. DPO with proper chosen/rejected pairs is simpler and more proven.

**Changes:**

1. Generate rejected responses using the base model (liberal-reformist defaults)
2. Use SFT teacher answers as chosen responses
3. Run DPO training with existing `train_dpo.py` infrastructure
4. Focus DPO pairs on the adversarial questions (Type E) where the model needs to suppress liberal defaults

**Risk:** Low. DPO infrastructure already exists and is tested.

**Expected outcome:** More reliable alignment than GRPO v1, but loses the multi-objective optimization and exploration benefits of GRPO.

## Recommended Path: Option A + Option B (combined)

Start with Option A (fix rewards) as the foundation, then add Option B (few-shot prompts) on top. This gives us:

1. **Concrete, non-zero reward signal** from keyword-based DM alignment check
2. **DM-aligned starting point** from few-shot prompts
3. **Diagnostic visibility** from per-component reward logging
4. **Low risk** — same infrastructure, same runtime budget

If this still fails to produce improvement after 100 steps, fall back to Option D (DPO).

## Implementation Plan

### Phase 1: Fix reward functions (Option A)

**File: `src/student/rewards.py`**

1. Add `compute_dm_keyword_alignment(text)` function:
   - Check for DM-specific analytical patterns (material conditions, structural causality, frame critique)
   - Return 0-1 score based on number of matched pattern categories
   - 3 pattern categories required for full score

2. Fix `compute_directional_assertion(text)`:
   - Remove negative penalty (hedging keywords shouldn't cancel positive matches)
   - Score based purely on positive matches: `min(1.0, positive_matches / 2)`

3. Update `compute_reward()` to support component logging:
   - Return dict with per-component scores instead of just total
   - Components: `{"dm_alignment": float, "directional": float, "total": float}`

**File: `src/student/grpo_config.py`**

4. Update reward weights:
   - `dm_alignment`: 0.65 (up from 0.50)
   - `directional_assertion`: 0.35 (up from 0.20)
   - Remove `format` and `length`

5. Add `reward_log_components: true` flag

### Phase 2: Add few-shot prompts (Option B)

**File: `src/student/train_grpo.py`**

6. Modify prompt construction to include 1-2 few-shot DM-aligned examples:
   - Load a small set of high-quality DM answers from teacher data
   - Prepend to each prompt before the target question
   - Use system prompt: "Your analytical frame examines material conditions and power structures."

7. Increase `max_completion_length` from 512 to 1024

### Phase 3: Add diagnostic logging

**File: `src/student/train_grpo.py`**

8. Update CSV logging to include per-component rewards:
   - Columns: `step, loss, avg_reward, dm_reward, dir_reward, elapsed_s, vram_gb`

### Phase 4: Smoke test

9. Run 10 steps with new rewards to verify:
   - Mean reward > 0.3 (above format/length floor)
   - DM alignment component > 0 on some batches
   - Directional assertion component > 0 on some batches
   - Training is stable (no NaN loss)

### Phase 5: Full training run

10. Run full 500-step training with new configuration
11. Monitor reward trajectory — expect upward trend over first 100-200 steps

## What NOT to Change

- Training loop architecture (custom GRPO, not TRL)
- LoRA configuration (r=16, alpha=16, same target modules)
- Learning rate (5e-7) or scheduler (cosine)
- Judge model (keep Qwen3.5-4B available as fallback if we want to A/B test keyword vs judge)
- Base model (same SFT checkpoint-330)
- Training data (same 1,500 questions)
